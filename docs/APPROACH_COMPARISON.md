# 디시인사이드 유동닉 댓글 자동 삭제 - 접근 방식 비교

> 작성일: 2026-03-07
> 프로젝트: dc_comment_protect

---

## 목적

디시인사이드 "특이점이 온다" 마이너 갤러리에서, 프록시 서버를 이용해 유동닉으로 도배하는 악성 댓글을 자동으로 삭제하는 프로그램 개발.

- **갤러리**: `thesingularity` (마이너 갤러리)
- **권한**: 부매니저 (댓글 삭제 권한 보유)
- **삭제 대상**: 유동닉 댓글 (닉네임 옆에 IP 주소가 표시되는 형태, 예: `ㅇㅇ(183.99)`)
- **범위**: 1~5페이지 순환

---

## 공통 기반: Chrome Extension

두 방식 모두 **크롬 확장 프로그램(Chrome Extension)** 형태로 구현한다.

### 이유
1. 이미 로그인된 브라우저 세션/쿠키를 그대로 사용 → 별도 로그인 불필요
2. 부매니저 권한이 쿠키에 포함되어 있으므로 권한 문제 없음
3. 토글 ON/OFF UI를 팝업으로 자연스럽게 구현 가능
4. 갤러리에 상주하면서 사용하는 패턴에 최적화

---

## 방식 A: DOM 페이지 이동 방식

### 개요
실제로 각 게시물 페이지로 이동(navigation)하여, DOM에서 댓글을 파싱하고, 체크박스를 클릭하고, 삭제 버튼을 누르는 방식.

### 동작 흐름
```
1. 갤러리 목록 페이지(1페이지) 로드
2. 게시물 목록에서 각 게시물 URL 파싱
3. 첫 번째 게시물 페이지로 이동 (window.location 또는 tab 이동)
4. 페이지 로드 완료 대기
5. DOM에서 댓글 영역 파싱
6. 유동닉(IP 포함) 댓글의 체크박스를 찾아 선택
7. 삭제 버튼 클릭
8. 삭제 완료 대기
9. 다음 게시물로 이동 → 3번으로
10. 현재 페이지 게시물 모두 완료 → 다음 페이지(2페이지)로
11. 5페이지까지 완료 → 1페이지로 돌아가서 반복
```

### 구현 방식
- `content.js`에서 현재 페이지의 DOM을 직접 조작
- 페이지 이동은 `chrome.tabs.update()` 또는 `window.location.href` 사용
- 각 페이지 로드 후 `content script`가 자동으로 실행되어 댓글 처리

### 속도 분석
| 단계 | 소요 시간 |
|------|----------|
| 게시물 페이지 로드 (full navigation) | ~1.5-3초 |
| DOM 파싱 + 체크박스 선택 | ~100-200ms |
| 삭제 버튼 클릭 + 응답 대기 | ~500ms-1초 |
| **게시물 1개당 합계** | **~2-4초** |

- 페이지당 게시물: ~50개 (기본 설정)
- 5페이지 총 게시물: ~250개
- **1사이클 예상 시간: ~10-15분**

### 장점
- 구현이 직관적이고 단순
- 실제 사용자 행동과 동일하므로 차단 위험 낮음
- 디버깅이 쉬움 (페이지가 보이므로)

### 단점
- **매우 느림** (페이지 렌더링이 병목)
- 사용 중 브라우저 탭이 계속 이동됨
- 브라우저에 부하가 큼
- 도배 대응 속도가 부족

## 방식 B: 내부 API 직접 호출 방식 ⭐ (채택)

### 개요
페이지 이동 없이, Chrome Extension의 `background.js`에서 `fetch()`를 사용하여 디시인사이드 내부 API를 직접 호출하는 방식. 브라우저의 로그인 쿠키가 자동으로 포함되므로 인증 문제 없음.

### 동작 흐름
```
1. fetch()로 갤러리 목록 API 호출 → 게시물 번호 목록 획득
2. 각 게시물에 대해 fetch()로 댓글 목록 API 호출
3. 응답에서 유동닉(IP 포함) 댓글 필터링 (메모리 내 처리)
4. 유동닉 댓글이 있으면 → fetch()로 댓글 삭제 API 호출
5. 다음 게시물로 → 2번으로
6. 현재 페이지 게시물 모두 완료 → 다음 페이지
7. 5페이지까지 완료 → 1페이지로 돌아가서 반복
```

### 구현 방식
- `background.js` (Service Worker)에서 모든 API 호출 수행
- `popup.js`에서 토글 ON/OFF 제어
- 페이지 이동 없음 → 백그라운드에서 모든 작업 수행
- 브라우저 쿠키 자동 포함 (`credentials: 'include'`)

### 속도 분석
| 단계 | 소요 시간 |
|------|----------|
| 게시물 목록 fetch (1페이지) | ~200-300ms |
| 댓글 목록 fetch (1게시물) | ~200-300ms |
| 유동닉 필터링 (메모리 내) | ~1ms |
| 댓글 삭제 fetch (1요청) | ~200-300ms |
| **게시물 1개당 합계** | **~0.5-1초** |

- 유동닉 댓글이 없는 게시물은 삭제 단계 스킵 → 더 빠름
- 안전 딜레이 포함 시: 게시물당 ~0.5-1초
- **1사이클 예상 시간: ~3-5분**

### 장점
- **10배 빠름** (페이지 렌더링 없음)
- 브라우저 UI에 영향 없음 (백그라운드 실행)
- 다른 작업 하면서도 동시 실행 가능
- 도배에 빠르게 대응 가능

### 단점
- 디시인사이드 내부 API 리버스 엔지니어링 필요
- API 변경 시 수정 필요
- 레이트 리밋/캡차 대응 필요

---

## 최종 비교표

| 항목 | 방식 A (DOM) | 방식 B (API) ⭐ |
|------|-------------|----------------|
| **속도** | ~10-15분/사이클 | ~3-5분/사이클 |
| **브라우저 영향** | 탭 이동됨 | 없음 (백그라운드) |
| **구현 난이도** | 쉬움 | 보통 |
| **차단 위험** | 낮음 | 보통 (딜레이로 완화) |
| **유지보수** | 쉬움 | API 변경 시 수정 필요 |
| **도배 대응** | 느림 🐢 | 빠름 🚀 |

---

## 결론

**방식 B (내부 API 직접 호출)** 채택.

악성 도배 공격의 속도를 고려하면, 방식 A의 10-15분 사이클로는 실시간 대응이 불가능. 방식 B의 3-5분 사이클이 현실적인 방어 수단.

추후 API 구조가 변경되면 방식 A로 폴백할 수 있도록, 방식 A의 구현도 별도 브랜치로 보관 가능.
