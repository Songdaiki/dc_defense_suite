## 3. TCP 연결이 실제로 성립한 증거

가장 중요한 시작 구간:

```text
2026-04-19 01:52:37.820290  1.210.3.152:64820 -> 121.138.132.127:1698  SYN
2026-04-19 01:52:37.825492  121.138.132.127:1698 -> 1.210.3.152:64820  SYN,ACK
2026-04-19 01:52:37.825568  1.210.3.152:64820 -> 121.138.132.127:1698  ACK
```

이 3줄의 의미:

- 클라이언트가 `1698/tcp` 로 접속 시도
- 서버가 응답
- TCP 3-way handshake 성립

즉, **이 서버는 실제로 살아 있고 `1698` 포트에서 클라이언트를 받는다.**

## 4. 첫 번째 소켓은 바로 끊겼고, 두 번째 소켓에서 세션이 이어짐

첫 번째 시도:

```text
01:52:37.836506  client -> server  length 965
01:52:37.842095  server -> client  RST
```

해석:

- 첫 번째 연결은 handshake 직후 클라이언트가 데이터를 보냈고
- 서버가 `RST` 로 바로 닫았다

하지만 바로 다음 두 번째 시도:

```text
01:52:37.842373  1.210.3.152:64822 -> 121.138.132.127:1698  SYN
01:52:37.847864  121.138.132.127:1698 -> 1.210.3.152:64822  SYN,ACK
01:52:37.847936  1.210.3.152:64822 -> 121.138.132.127:1698  ACK
01:52:37.867707  client -> server  length 439
01:52:37.880116  server -> client  length 1460
01:52:37.880116  server -> client  length 1460
01:52:37.880116  server -> client  length 217
```

그리고 이후에도 수 초간 양방향 데이터가 계속 오갔다.

의미:

- 공식 클라이언트는 첫 소켓이 닫혀도 바로 재시도한다
- 두 번째 소켓에서는 **한두 패킷이 아니라 연속적인 세션 데이터가 오간다**
- 따라서 이번 건은 단순 포트 오픈 확인 수준이 아니라 **실제 프로토콜 협상 단계까지 진입한 케이스**다

쉽게 예시로 말하면:

- 첫 통화는 받자마자 끊김
- 두 번째 통화는 실제 대화가 길게 이어짐

## 5. UDP 보조 트래픽도 같이 보임

같은 캡처에서 아래도 관측됐다.

```text
121.138.132.127:40000 -> 1.210.3.152:61402  UDP
121.138.132.127:40001 -> 1.210.3.152:59100  UDP
```

이번 단계에서 중요한 건 TCP `1698` 이고, UDP는 보조 채널 또는 추가 트래픽일 가능성이 있다.

핵심은:

- **성공 여부 판단 1순위는 `1698/tcp` handshake + 지속 데이터**
- UDP 해석은 그 다음 문제

## 6. 이번 성공 케이스에서 확정된 사실

- raw feed의 `IP=121.138.132.127` 와 `SslPorts=1698` 는 실제 연결 대상 정보와 일치한다
- 공식 SoftEther VPN Gate 플러그인은 이 raw 값을 사용해 실제 세션을 연다
- `1698/tcp` 는 실제로 접속 가능했고 서버가 응답했다
- 첫 번째 소켓은 리셋됐지만, 두 번째 소켓에서 양방향 세션이 이어졌다
- 따라서 **"raw 1개 -> 실제 연결 시도"는 가능**하다는 게 네트워크 레벨에서 확인됐다

## 7. 아직 이 문서만으로는 확정 못 하는 것

이번 문서는 네트워크 캡처 문서다.
