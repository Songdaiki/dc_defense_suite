# VPNGate 공식 200개 Feed 10분 새로고침 가이드

## 한 줄 요약

공식 VPNGate 200개 feed는 **10분마다 한 번만 새로고침**하고,  
매번 **전체 스냅샷은 저장하되**, 실제 반영은 **이전 스냅샷과의 delta만 본다**.

쉽게 예시로 말하면:

- 23:00에 200개 목록 전체 저장
- 23:10에 다시 200개 목록 전체 저장
- 두 목록을 비교해서
  - 새로 생긴 IP
  - 사라진 IP
  - 그대로 있는 IP
  만 계산한다

즉, 핵심은 아래 두 가지다.

- 저장: **매번 전체 스냅샷 저장**
- 비교/반영: **직전 스냅샷과의 차이만 사용**

## 왜 10분인가

이번 측정 결과를 보면, feed는 매 5분마다 조금씩 흔들리는 게 아니라  
**몇 구간은 그대로 있다가 특정 시점에 한 번에 많이 바뀌는 패턴**에 더 가까웠다.

쉽게 말하면:

- 어떤 5분 구간은 `변화 0`
- 다음 5분 구간도 `변화 0`
- 그런데 그 다음 구간에서 한 번에 크게 바뀜

그래서 운영 기준으로는:

- `5분마다 무조건 새로고침`: 너무 잦음
- `10분마다 새로고침`: 요청 수를 줄이면서도 변화 구간은 충분히 잡음

즉, 이번 판단은 **정확히 10분마다 교체된다**는 뜻이 아니라,
**운영용 polling 주기는 10분으로 잡는 게 적당하다**는 뜻이다.

## 이번 측정에서 확인한 변화 패턴

30분 샘플 기준:

- 전체 고유 IP 총합: 처음부터 끝까지 보면 `199` 기준에서 크게 바뀌는 구간이 한 번 있었다
- KR 고유 IP는 `60 -> 63 -> 63 -> 65 -> 65 -> 60 -> 60`
- KR 변화량도 `0인 구간`과 `크게 바뀐 구간`이 섞여 있었다

KR만 쉽게 보면:

- `1->2`: 조금 바뀜
- `2->3`: 그대로
- `3->4`: 조금 바뀜
- `4->5`: 그대로
- `5->6`: 크게 바뀜
- `6->7`: 그대로

이 패턴이면 실무적으로는 **5분마다 계속 때리는 것보다 10분마다 보는 쪽이 낫다**.

## 운영 기준

### 1. 데이터 소스

공식 feed 원천은 아래다.

```text
http://xd.x1.client.api.vpngate2.jp/api/?session_id=...
```

여기서 raw DAT를 받고, 저장소 스크립트로 복호하면 공식 클라이언트가 받는 200개 목록이 나온다.

관련 스크립트:

- `scripts/decode-vpngate-official-dat.mjs`
- `scripts/measure-vpngate-feed-changes.mjs`

## 2. 새로고침 주기

운영 기준:

- **10분마다 1회 새로고침**

권장 흐름:

1. 현재 공식 200개 feed를 받는다.
2. raw DAT / decoded JSON / unique IP 목록을 **전체 스냅샷으로 저장**한다.
3. 이전 스냅샷이 있으면 그때만 delta를 만든다.
4. 새로 생긴 IP만 추가 반영하고, 사라진 IP만 제거 검토한다.

즉:

- 저장은 매번 200개 전체를 남기고
- 실제 반영은 **직전 스냅샷 대비 변화량만 본다**

## 3. 왜 delta만 보면 되나

예시:

- 23:00 목록 KR IP가 `60개`
- 23:10 목록 KR IP가 `64개`
- 비교 결과:
  - 새로 생김 `5`
  - 사라짐 `1`
  - 유지 `59`

이 경우 저장은 이렇게 된다.

- `23:00 전체 스냅샷 저장`
- `23:10 전체 스냅샷 저장`

그 다음 실제로 새로 처리할 대상은:

- 새로 생긴 `5개`
- 사라진 `1개`

뿐이다.

즉, **저장은 전체**, **반영은 delta**다.

## 4. 10분 주기 실행 예시

### 30분 측정

```bash
node scripts/measure-vpngate-feed-changes.mjs \
  --duration-minutes 30 \
  --interval-minutes 10
```

이렇게 하면 스냅샷은 보통 아래처럼 생긴다.

- `t=0`
- `t=10`
- `t=20`
- `t=30`

즉 총 `4개 스냅샷`이다.

### KR만 따로 보고 싶을 때

```bash
node scripts/measure-vpngate-feed-changes.mjs \
  --duration-minutes 30 \
  --interval-minutes 10 \
  --country-short KR
```

이 경우 콘솔에서 KR 변화량도 같이 바로 볼 수 있다.

예시:

```text
[1/4] 완료: hosts=200, unique_ips=199, duplicates=1, KR_hosts=60, KR_unique_ips=60
[2/4] 변화량: 새로 생김=7, 사라짐=7, 유지=192, KR_새로생김=4, KR_사라짐=1, KR_유지=59
```

해석:

- 전체 200개 기준으로는 `7개 추가 / 7개 제거`
- 한국 KR만 보면 `4개 추가 / 1개 제거`

## 5. 실제 운영에 적용할 때의 기준

운영 문구로 정리하면 아래처럼 보면 된다.

- feed 새로고침: `10분마다 1회`
- 처리 단위: `직전 스냅샷 대비 delta`
- 우선 확인 대상: `KR IP 변화`, 필요시 전체 국가 변화

쉽게 예시로 말하면:

- 23:00 전체 스냅샷 저장
- 23:10 전체 스냅샷 저장
- 23:10에서 `23:00 -> 23:10 delta` 계산
- 23:10에 새로 생긴 KR IP만 overlay에 추가
- 23:10에 사라진 KR IP만 제거 검토
- 23:20에도 다시 전체 스냅샷 저장 후 delta 계산

## 6. 주의할 점

이 문서의 10분 기준은 **공식 200개 feed를 받아오는 polling 기준**이다.

이건 아래와는 다르다.

- 실제 VPN 연결 빈도
- 공격 감지 주기
- 차단 반영 주기

즉:

- `feed 정보 수집`은 10분마다
- 그 feed를 실제 방어 로직에 어떻게 반영할지는 별도 정책으로 나눠서 봐야 한다

## 7. 결론

이번 기준은 아래 한 문장으로 정리된다.

**공식 VPNGate 200개 feed는 10분마다 새로고침하고, 전체 스냅샷은 매번 저장하되 실제 반영은 delta만 처리한다.**

쉽게 예시로 끝내면:

- 예전 방식: `5분마다 전체 200개 다시 봄`
- 지금 기준: `10분마다 전체 스냅샷 저장 + 비교는 delta만 사용`

후자가 요청 수도 적고, 실제 변화 패턴에도 더 맞는다.
