## 한줄 결론

이 샘플은 `매번 완전히 새로운 프록시`를 쓰는 형태라기보다, `여러 대역을 반복적으로 돌려 쓰는 형태`에 가깝다.

다만 디시 기록에는 `A.B.*.*`까지만 보이므로, 실제 고유 IP 개수나 실제 프록시 서버 개수는 확정할 수 없다.

## 핵심 수치

- 총 IP 표기 횟수: `2350`
- 고유한 가시 prefix 수: `152`
- 형식: `A.B.*.*`

즉 현재 보이는 범위에서만 보면:

- 2350건 전체가 전부 다른 대역은 아님
- `152개`의 visible prefix를 반복해서 사용

## 관찰 포인트

### 1. 완전 랜덤 글로벌 풀처럼 보이지는 않는다

`A.B.*.*` 기준만 봐도 반복도가 높다.

- 고유 prefix `152`
- 총 등장 `2350`
- 상위 50개 prefix가 전체의 `60%` 이상

즉 소수 핵심 대역을 반복적으로 재사용하는 흔적이 강하다.

### 2. 하지만 "프록시 서버 개수"는 아직 계산할 수 없다

현재 로그는 `A.B.*.*`까지만 보인다.

예를 들어 `121.140.*.*` 한 줄 안에는 실제로:

- 수십 개 IP가 있을 수도 있고
- 수백 개 IP가 있을 수도 있고
- 같은 소수 IP를 반복 사용 중일 수도 있다

따라서 현재 자료로 확정 가능한 것은:

- `가시 prefix pool` 크기
- `반복 사용 빈도`

확정 불가능한 것은:

- 실제 고유 IP 개수
- 실제 프록시 노드 개수

### 3. 현재 데이터만으로 가능한 실전 판단

현재 시점에서 방어 관점으로 쓸 수 있는 판단은 다음이다.

- 적어도 `152개 visible prefix` 수준의 회전 흔적이 있다
- 그중에서도 자주 나오는 대역은 우선 감시/차단 후보로 볼 수 있다
- 완전히 새 대역만 계속 나오는 공격보다, 재등장 네트워크가 꽤 많다

## 현재 자료의 한계

- `A.B.*.*`까지만 보여서 실제 IP 수는 모름
- 서로 다른 게시글/댓글이 같은 실제 IP인지도 모름
- 통신사/IDC/프록시 사업자 정보는 아직 붙이지 않음

## 다음 단계 제안

### 1. visible prefix 기준 블랙리스트 후보 만들기

예:

- 20회 이상 등장한 `A.B.*.*`만 별도 분리

### 2. 시간대 결합

현재 파일에 시간 정보가 있으므로:

- 같은 분/초에 대량으로 튀는 prefix
- 짧은 시간에 순환하는 묶음

을 보면 회전 패턴이 더 선명해진다.

### 3. 차단 로그와 교차분석

운영 차단 후에도 재등장하는 prefix를 보면:

- 차단 효과가 없는 대역
- 계속 돌려쓰는 핵심 풀

을 분리할 수 있다.

## 결론

이 자료만 보면 공격자는 `아주 큰 완전 무작위 풀`보다는, `반복 등장하는 다수의 prefix 묶음`을 돌려 쓰고 있다.

따라서 방어 전략도:

- 개별 게시물 대응만 하는 것보다
- 자주 반복되는 `A.B.*.*` 묶음을 기준으로 감시/차단 우선순위를 두는 쪽이 더 효율적이다.
