# IP Ban Spec

## 목적

`dc_post_protect`와 비슷하게 게시물 목록을 순회하면서 도배성 게시물을 찾고, 선택된 게시물 기준으로 운영 차원의 차단 API를 호출하는 확장 프로그램을 만든다.

이 프로젝트의 핵심 목표는 다음이다.

- IP 유동 도배 게시물을 빠르게 운영 차단한다.
- 필요하면 삭제 옵션을 함께 쓸 수 있게 한다.
- 가능하면 식별코드가 있는 유저도 `식별 코드 + IP` 형태로 함께 차단할 수 있게 한다.

## 2026-03-07 기준 확인된 사실

### 1. `이용자 직접 차단` 팝업은 IP 유동용이 아니다

직접 차단 팝업의 `식별 코드` 입력칸은 아래 조건을 강제한다.

- `5~20자`
- `영문자/숫자 조합`

따라서 아래 값들은 직접 차단 팝업에 넣을 수 없다.

- `ㅇㅇ(112.144.*.*)`
- `112.144.*.*`
- 순수 IP 문자열

결론:

- `이용자 직접 차단` = 식별코드(`uid`) 기반 차단용
- IP 유동 직접 입력용 아님

### 2. 리스트 작성자 정보는 `uid`와 `ip`가 분리되어 있다

실목록 HTML 확인 기준:

- 식별코드 있는 대상: `data-uid="mental7264"`, `data-ip=""`
- IP 유동 대상: `data-uid=""`, `data-ip="112.157"`

즉 IP 유동은 브라우저 레벨에서 식별코드가 비어 있다.

결론:

- IP 유동을 식별코드 입력 방식으로 우회 차단하는 설계는 버린다.

### 3. 작성자 메뉴의 `차단 IP`는 운영 제재가 아니다

작성자 우클릭/작성자 메뉴에서 보이는:

- `작성글 검색`
- `동일 IP 선택`
- `차단 닉네임`
- `차단 IP`

이 메뉴의 `차단 IP`는 운영 차단이 아니라 개인 차단이다.

공개 스크립트 기준 흐름:

- `common.js` -> `ub_save_conf_one(..., 'ip', layer_user_ip)`
- `user_block.js` -> `localStorage`에 저장 후 `location.reload()`

결론:

- 이 경로는 다른 유저에게 영향을 주지 않는다.
- 자동 운영 방어 프로그램의 대상 API가 아니다.

### 4. 실제 운영 차단 경로는 `게시물 선택 -> 차단`이다

관리자 리스트에서 게시물을 체크한 뒤 상단 `차단` 버튼을 누르면 운영 차단 팝업이 열린다.

이 팝업에서 확인 가능한 옵션:

- 차단 기간
- 차단 사유
- `삭제`
- `식별 코드의 IP 차단`

이 경로가 실제 운영 차단 API와 연결된다.

### 5. 실제 관리자 차단 API를 확인했다

관찰된 요청:

- Method: `POST`
- URL: `https://gall.dcinside.com/ajax/minor_manager_board_ajax/update_avoid_list`

관찰된 Form Data:

```text
ci_t=***
id=thesingularity
nos[]=1040937
parent=
avoid_hour=1
avoid_reason=4
avoid_reason_txt=
del_chk=0
_GALLTYPE_=M
avoid_type_chk=1
```

관찰된 Response:

```json
[{"result":"success","msg":"..."}]
```

`msg`는 유니코드 이스케이프가 섞인 한국어 메시지였다. 현재까지는 `result: success` 확인이 핵심이다.

### 6. `도배기` 탭은 `search_head=130`으로 필터링된다

확인 페이지:

- `https://gall.dcinside.com/mgallery/board/lists/?id=thesingularity&sort_type=N&search_head=130&page=1`

확인 내용:

- hidden input `search_head=130`
- 탭 링크 `listSearchHead(130)` 라벨이 `도배기`
- 목록 `gall_subject`에 `도배기`가 연속적으로 표시됨

결론:

- 현재 대상 갤(`thesingularity`)에서는 `130`이 `도배기` head id다
- 따라서 `도배기 탭 순회 = search_head=130&page=n` 방식으로 구현 가능하다

주의:

- `130`이 모든 갤러리의 공통값이라고 단정하면 안 된다
- 갤별 설정값일 가능성이 있으므로 설정으로 분리하는 편이 안전하다

### 7. 일반 관리자 차단과 `특정 통신사 IP 대역 제한`은 다른 기능으로 봐야 한다

현재까지의 근거:

- 일반 관리자 차단 API는 `nos[]`를 보내고 서버가 대상 글 작성자를 기준으로 차단 처리한다
- 일반 관리자 차단 요청에는 `A.B.*.*` 같은 가시 IP prefix가 직접 들어가지 않는다
- 반면 `특정 통신사 IP 대역 제한`은 별도 기능으로 존재하고, 최대 개수 제한도 따로 있다

현재 가장 타당한 해석:

- 일반 관리자 차단 = 실제 IP 또는 더 좁은 서버 내부 식별 단위 차단
- 특정 통신사 IP 대역 제한 = 진짜 범위 차단

주의:

- 이 구분은 현재까지 확보한 요청/UI 구조를 바탕으로 한 추론이다
- 실서버 DB 값을 직접 본 것은 아니므로 100% 증명은 아님

## 파라미터 해석

아래 해석은 현재까지 확인된 UI 문맥 + 실제 요청 기준이다.

### `ci_t`

- `ci_c` 쿠키에서 가져오는 관리자 요청 토큰
- 필수
- 로그/문서에 평문 저장 금지

### `id`

- 갤러리 ID
- 예: `thesingularity`

### `nos[]`

- 차단 대상 게시물 번호 배열
- 단건도 배열 키를 사용한다
- 다건 배치 가능성이 높다

### `parent`

- 현재 캡처에서는 빈 값
- 의미 미확정
- 댓글/답글 차단과 공용 파라미터일 가능성 있음

### `avoid_hour`

- 차단 기간 값
- 관찰값 `1`
- UI의 `1시간 / 6시간 / 1일 / 7일 / 14일 / 31일`과 매핑됨
- 정확한 코드표는 추가 확인 필요

### `avoid_reason`

- 차단 사유 코드
- 관찰값 `4`
- UI의 `음란성 / 광고 / 욕설 / 도배 / 혐오 콘텐츠 / 저작권 침해 / 명예훼손 / 직접 입력`과 매핑됨
- 정확한 코드표는 추가 확인 필요

### `avoid_reason_txt`

- 직접 입력 사유 텍스트
- 선택 사유 사용 시 빈 문자열 가능

### `del_chk`

- 삭제 여부
- 관찰값 `0`
- 추정:
  - `0` = 삭제 안 함
  - `1` = 삭제 함께 수행

### `_GALLTYPE_`

- 갤러리 타입
- 관찰값 `M`
- 마이너 갤러리 기준

### `avoid_type_chk`

- 현재 가장 중요한 옵션
- 관찰값 `1`
- UI 문맥상 `식별 코드의 IP 차단` 체크값으로 해석됨

현재 해석:

- `0` = 일반 차단
- `1` = IP까지 함께 묶는 차단

주의:

- 순수 IP 유동에서 이 값이 어떤 의미로 처리되는지는 추가 확인 필요
- 식별코드 보유 사용자에서 `(uid) + IP` 형태를 만드는 플래그일 가능성이 높다

추가 실험 결과:

- 순수 IP 유동 대상으로 `avoid_type_chk=1` 상태에서 차단했을 때 관리내역은 `닉네임(IP)`로 기록되었다
- 같은 순수 IP 유동 대상으로 체크를 해제한 상태에서도 관리내역은 동일하게 `닉네임(IP)`로 기록되었다

현재 결론:

- 순수 IP 유동에게는 `avoid_type_chk`가 의미 없거나 서버에서 무시될 가능성이 높다
- 이 플래그는 식별코드가 있는 대상에게만 의미가 있을 가능성이 높다

## 타입별 기대 동작

### A. 식별코드가 있는 사용자

예:

- `(semester0653) + IP`
- `(which4294) + IP`

예상 동작:

- 일반 차단: 식별코드 기반 차단
- `avoid_type_chk=1`: 식별코드 + IP 동시 차단

### B. 순수 IP 유동

예:

- `(121.241.*.*)`
- `(115.41.*.*)`
- `(222.113.*.*)`

확정된 사실:

- 직접 차단 팝업의 식별코드 입력으로는 차단 불가
- `게시물 선택 -> 차단` 경로를 통해서는 운영 차단 가능
- 관리내역에는 `닉네임(IP)` 형태로 남는다
- `avoid_type_chk`를 켜도/꺼도 결과 표기는 동일했다

현재 가장 그럴듯한 서버 동작:

- 클라이언트는 `nos[]`만 보낸다
- 서버가 선택된 게시물 작성자의 실제 정보를 조회한다
- 식별코드가 없으면 IP 기준 차단으로 처리한다

이 마지막 3줄은 현재까지의 관찰을 기반으로 한 추론이다.

## 관리내역 표기 규칙

공개 관리내역 기준으로 최소 두 종류가 존재한다.

- `닉네임(식별 코드) + IP`
- `닉네임(IP)`

해석:

- `+ IP` = 식별코드 차단에 IP를 함께 묶은 타입
- `닉네임(IP)` = 순수 IP 기준 차단 타입

## 자동화 가능성

현재까지는 자동화 가능성이 높다.

이유:

- 차단 API endpoint가 확인됨
- 대상 식별에 raw IP 직접 입력이 필요하지 않음
- `nos[]` 배열로 게시물 번호만 넘기면 서버가 차단 처리할 가능성이 높음
- 구조가 `dc_post_protect`와 매우 비슷함

예상 자동화 흐름:

1. 목록 페이지 순회
2. 도배 패턴/IP 유동 글 탐지
3. 차단 대상 게시물 번호 수집
4. `update_avoid_list` 호출
5. 성공/실패 로깅

## 제품 플로우

현재 목표 플로우는 아래와 같다.

1. `도배기` 탭 페이지를 순회한다
2. 목록에서 유동 작성 게시물만 수집한다
3. 수집된 게시물 번호를 기준으로 `1시간 차단`을 건다
4. 성공한 차단 내역은 로컬 로그에 저장한다
5. 공격 종료 후, 사용자가 수동으로 `해제 토글/버튼`을 눌렀을 때만 이 프로그램이 기록한 차단 내역 해제를 시도한다

### 차단 플로우

권장 기본값:

- `search_head=130`
- `avoid_hour=1`
- `avoid_reason=도배`
- `del_chk=0`

예상 실행 순서:

1. `search_head=130&page=n` 목록 조회
2. `공지`, `설문`, 고정글, 비유동 작성 글 스킵
3. 유동 게시물 `postNo` 수집
4. `update_avoid_list` 호출
5. 응답 `result=success` 확인
6. 로컬 로그 저장

### 로그 저장 플로우

차단 후 나중에 해제를 하려면 최소한 아래 정보가 필요하다.

- `postNo`
- `galleryId`
- `bannedAt`
- `avoidHour`
- `avoidReason`
- `delChk`
- `runId`
- 가능하면 `writerKey` 예: `ㅇㅇ|211.233`
- 가능하면 당시 보였던 `닉네임(IP)` 또는 `닉네임(식별 코드/IP)` 표기

### 해제 플로우

현재 방향:

- 자동 해제 스케줄러는 만들지 않는다
- 운영자가 직접 `해제 토글/버튼`을 눌렀을 때만 해제 프로세스를 실행한다

이유:

- 공격 종료 판단을 프로그램이 자동으로 하기보다 운영자가 판단하는 편이 안전하다

예상 UX:

1. 평소에는 자동 차단만 동작
2. 공격 종료 후 운영자가 팝업에서 `내가 차단한 대상 해제` 버튼 클릭
3. 저장된 로그에서 `postNo`, `writerKey`, `runId`를 읽는다
4. `https://gall.dcinside.com/mgallery/management/block?id=<galleryId>` 차단 목록 1페이지부터 읽는다
5. 각 row에서 게시물 링크의 글번호와 `onclick="set_avoid(this, 'R', '해제ID', 0)"`를 파싱한다
6. 로컬 로그의 `postNo`와 일치하는 row를 찾으면 `해제ID`를 추출한다
7. 아직 못 찾은 대상이 있으면 2페이지, 3페이지 식으로 다음 페이지를 계속 읽는다
8. 필요한 대상을 모두 찾으면 스캔을 즉시 중단한다
9. `POST /ajax/managements_ajax/set_avoid`에 `avoid_type=R`, `nos[]=해제ID`로 호출한다
10. 성공/실패 결과를 다시 로그에 남김

### 해제 페이지 순회 전략

차단 목록은 페이지가 여러 장일 수 있으므로, 해제는 단일 페이지 전제 없이 설계해야 한다.

기본 전략:

1. 최신순 기준으로 1페이지부터 읽는다
2. 각 페이지 row를 파싱해 `postNo -> 해제ID` 매핑을 만든다
3. 이번 세션(`runId`)에 차단한 `postNo`들을 하나씩 매칭한다
4. 아직 못 찾은 대상이 남아 있으면 다음 페이지로 이동한다
5. 전부 찾으면 즉시 종료한다

중단 조건:

- 이번 세션의 대상 전부를 찾았을 때
- row의 처리 날짜/시간이 `bannedAt`보다 충분히 오래된 구간으로 내려갔을 때
- 최대 스캔 페이지 수를 넘겼을 때

실전 해석:

- 해제는 자동 방어 루프가 아니라 수동 실행이므로, 차단만큼 초고속일 필요는 없다
- 대신 1페이지에 없다고 놓치면 안 되므로, 최근 페이지부터 순차 스캔하는 쪽이 맞다
- 보통은 이번 세션 차단이 최근 내역에 몰리므로 몇 페이지 안에서 끝날 가능성이 높다

## 지금 구현 가능한 범위

현재 스펙만으로 바로 구현 가능한 부분:

- `도배기 탭(search_head=130)` 페이지 순회
- 목록 파싱
- 유동 게시물 판별
- 관리자 차단 API 호출
- `1시간 차단` 기본값 적용
- 성공/실패 로그 저장
- 같은 게시물 중복 차단 방지
- 수동 해제를 위한 로컬 로그 기반 대상 추적
- 차단 목록 HTML 파싱을 통한 수동 해제

즉, `자동 차단 + 수동 해제`까지 지금 바로 구현 가능한 수준이다.

## 아직 미확정인 부분

### 1. 차단 해제 API

확인된 것:

- Endpoint: `POST https://gall.dcinside.com/ajax/managements_ajax/set_avoid`
- Response: `{"result":"success"}`
- 관찰된 Payload:

```text
ci_t=***
gallery_id=thesingularity
_GALLTYPE_=M
avoid_type=R
nos[]=10008103
ano=0
```

현재 해석:

- `avoid_type=R`는 해제(Release) 타입으로 보인다
- `nos[]`는 해제 대상 번호 배열이다
- `gallery_id`, `_GALLTYPE_`, `ci_t`가 필요하다
- 단건 해제는 이미 가능해 보이며, 배열 구조상 다건 해제 가능성도 높다

실측 확인:

- 테스트 게시물 번호: `1026097`
- 화면에 보인 차단 번호: `11127`
- 실제 해제 요청 payload: `nos[]=10008138`
- 추가 row HTML 확인:

```html
<td class="blocknum" data-num="10008485">11149</td>
<td class="blockcontent">
  <span><em>게시글</em>
    <a href="//gall.dcinside.com/mgallery/thesingularity/1024419" target="_blank">갤 뭐임?</a>
  </span>
</td>
<td class="blockstate txtbtn">
  차단 중
  <button type="button" class="btn_blue_round small" onclick="set_avoid(this, 'R', '10008484', 0)">해제</button>
</td>
```

따라서:

- 해제 API의 `nos[]`는 게시물 번호가 아니다
- 해제 API의 `nos[]`는 화면에 보이는 짧은 차단 번호와도 다르다
- `nos[]`는 차단 내역의 별도 내부 식별값일 가능성이 높다
- 차단 목록 row 안에서도 `blocknum[data-num]` 값과 해제 버튼 `onclick`의 ID가 서로 다를 수 있다
- 실제 해제 요청에 써야 하는 값은 `onclick="set_avoid(..., '해제ID', ...)"`의 세 번째 인자다

현재 결론:

- `차단`은 구현 가능
- `수동 해제`도 endpoint 수준에서는 구현 가능하다
- 해제용 내부 ID는 `차단 목록 row HTML`에서 확보 가능하다
- 따라서 `수동 해제`는 `차단 목록 파싱 -> 해제 버튼 onclick 인자 추출 -> set_avoid 호출` 흐름으로 구현 가능하다
- 2026-03-07 기준 공개 JS(`common.js`, `user_block.js`, `mgallery.js`, `total_singo.js`)와 공개 페이지에서는 이 endpoint를 직접 찾지 못했고, 실제 관리자 화면 `Network` 캡처로 확정했다

### 1-1. 모바일 관리 스크립트에서 발견한 해제 단서

공개 모바일 관리 스크립트 `https://m.dcinside.com/js/management.min.js?v=251210`에서는 아래 함수가 확인되었다.

```javascript
function setAvoid(type, num, button) {
  var payload = { type: type, num: num };
  $.ajax({
    type: "POST",
    url: location.path,
    data: payload,
    dataType: "json"
  });
}
```

추가 단서:

- 다건 해제 시 `num`은 배열로 묶인다
- 모바일 UI 문자열에 `차단 해제하시겠습니까?`, `해제됨`, `재차단`이 존재한다
- 같은 스크립트의 `reAvoid()`는 `avoid_select` 화면으로 폼 POST를 보낸다

현재 해석:

- 모바일 관리 화면에는 실제 해제 기능이 존재한다
- 해제 요청 payload 핵심은 최소 `type`, `num`일 가능성이 높다
- 다만 위 코드에 `location.path`가 보이므로, 실제 요청 URL은 브라우저 Network로 최종 확인이 필요하다

추가 확인 결과:

- 실제 데스크톱 관리자 화면에서는 `POST /ajax/managements_ajax/set_avoid`가 호출되었다
- 따라서 모바일 관리 스크립트의 해제 흐름과 데스크톱 관리자 실제 호출이 같은 기능을 가리키는 것으로 보인다

### 2. 차단 목록 조회 API 또는 관리 화면 구조

해제를 하려면 아래 중 하나가 필요하다.

- 차단 목록 조회 API
- 차단 목록 HTML 파싱 경로
- 차단 시점에 받은 고유 식별값
- 또는 차단 내역의 `nos[]` 대상 번호를 얻는 안정적인 경로

현재 상태:

- 차단 목록 HTML 파싱 경로는 확보됐다
- `https://gall.dcinside.com/mgallery/management/block?id=...`의 row HTML에서
  게시물 링크와 해제 버튼 `onclick`을 같이 읽을 수 있다
- 페이지네이션이 있으므로 `page=n` 기준 순차 탐색이 필요하다

권장 매칭 방식:

1. 차단 목록 페이지 HTML을 1페이지부터 로드
2. 각 row에서 게시물 링크의 글번호를 추출
3. 우리가 저장한 `postNo`와 일치하는 row를 찾음
4. 해당 row의 `onclick="set_avoid(this, 'R', '10008484', 0)"`에서 세 번째 인자를 추출
5. 못 찾은 대상이 남아 있으면 다음 페이지로 넘어감
6. 필요한 대상을 모두 찾으면 그 값을 `nos[]`로 넣어 해제 API 호출

중요:

- 현재 해제 API의 `nos[]`는 게시물 번호도, 화면 차단 번호도 아니다
- 하지만 차단 목록 row HTML에서 해제용 내부 ID를 뽑을 수 있으므로 구현은 가능하다
- 현재 차단 로그에 최소 `postNo`, `galleryId`, `bannedAt` 정도를 저장해두면 목록 파싱 후 매칭이 가능하다
- `data-num`보다 해제 버튼 `onclick`의 세 번째 인자를 우선 신뢰한다
- 1페이지에 없을 수 있으므로 페이지 순회가 필수다

### 3. 기간/사유 코드 전체 매핑

현재 확인된 값:

- `avoid_hour=1`
- `avoid_reason=4`

아직 미확정:

- `6시간 / 1일 / 7일 / 14일 / 31일` 코드

## 현재 결론

이 프로젝트는 `raw IP를 직접 입력해서 차단하는 도구`가 아니라, `게시물 번호를 근거로 서버 차단 API를 호출하는 도구`로 설계해야 한다.

즉 올바른 방향은 아래다.

- 잘못된 방향: `IP 문자열 수집 -> 직접 차단 입력`
- 올바른 방향: `도배 게시물 감지 -> 게시물 번호 기준 운영 차단 API 호출`

현 시점 구현 우선순위는 아래다.

1. `도배기 탭 -> 유동 글 -> 1시간 차단` 자동화 먼저 구현
2. 차단 내역 로깅
3. 수동 해제 버튼을 위한 상태 저장 구조 마련
4. 차단 목록 파싱 기반 수동 해제 기능 추가
