# 한자 풀 추출기 요구사항 / 구현 스펙

## 목적

디시 관리기록 페이지(차단/해제 기록) 여러 페이지를 순회하면서, **제목에 들어간 한자/CJK 글자만** 한 글자씩 모두 수집한 뒤,  
그중 **중복해서 등장한 한자만** 추려서 쉼표(`,`)로 이어진 문자열로 출력하는 보조 확장을 만든다.

이 도구의 목표는 “중국어/한자형 도배 공격에서 반복적으로 재사용되는 한자 풀”을 빠르게 뽑아내는 것이다.

---

## 사용자가 원하는 최종 동작

사용자가 관리기록 페이지에서 확장을 연 뒤:

1. 시작 페이지와 끝 페이지를 정한다.
2. 예:
   - 시작 페이지: `1`
   - 끝 페이지: `400`
3. 실행 버튼을 누른다.
4. `최소 반복 횟수`를 정한다.
5. 페이지 범위와 최소 반복 횟수는 다음 실행을 위해 저장한다.
6. 확장은 각 페이지를 순회하면서 **제목만** 읽는다.
7. 제목 안에서 **한자/CJK 글자만** 한 글자씩 추출한다.
8. 모든 페이지에서 추출한 글자를 합친다.
9. 그중 설정한 횟수 이상 등장한 한자만 뽑는다.
10. 결과를 아래처럼 보여준다.

```text
刊,扺,縐,倻,偶,再,...
```

즉 사용자가 원하는 최종 결과는:
- **제목 기준**
- **한자 한 글자씩**
- **중복되는 글자만**
- **쉼표로 이어서**
보는 것이다.

### 아주 쉽게 풀어쓴 흐름

사용자가 원하는 건 정확히 이 흐름이다.

1. `1페이지` 제목을 읽는다.
2. 제목에 들어간 한자만 **한 글자씩** 뽑는다.
3. `2페이지`도 똑같이 읽는다.
4. 이 작업을 `400페이지`까지 반복한다.
5. 모은 한자를 전부 한데 합친다.
6. 그중 설정한 `최소 반복 횟수` 이상 나온 한자만 남긴다.
7. 결과를 아래처럼 보여준다.

```text
字,漢,龍,鳳,...
```

즉 이 도구는:
- 제목을 통째로 비교하는 도구가 아니라
- 제목 안의 **한자 낱글자**를 전부 모은 뒤
- **반복 출현한 글자 목록**만 뽑는 도구다.

### 현재 확정된 요구사항

이 문서 기준 1차 구현은 아래 요구사항으로 고정한다.

- 관리기록 `1페이지 ~ 400페이지`
- `게시글` 행만 대상
- `제목`만 분석
- 제목에서 `Han/CJK` 글자만 추출
- 모든 페이지 결과를 전체 누적
- `최소 반복 횟수` 이상 나온 글자만 포함
- **빈도 높은 순**으로 정렬
- 최종 결과는 `한자, 한자, 한자, ...` 형태로 출력
- 기본값은 `2회 이상`

## 분석 대상

### 포함
- 관리기록 페이지의 각 row 중 `게시글` row
- 각 row의 **제목(subject)** 텍스트

### 제외
- 닉네임
- IP
- 사유
- 날짜
- 시간
- 상태(차단 중 / 해제됨)
- 버튼 텍스트

즉 이 도구는 **게시글 제목만 본다.**

---

## 페이지 수집 범위

1차 구현 목표:
- `1페이지`부터 `400페이지`까지

즉 1차는 우선 아래 범위를 기본 전제로 한다.
- 시작 페이지: `1`
- 끝 페이지: `400`
- 마지막으로 실행한 시작/끝 페이지는 저장해서 다음 실행 때 그대로 보여준다.

추후 필요하면 사용자가 범위를 바꿀 수 있게 확장할 수 있다.

---

## 한자/CJK 판정 기준

이 도구는 “중국어 문장 해석”을 하지 않는다.  
그냥 **글자 종류**만 본다.

추출 대상:
- `CJK Unified Ideograph`
- `CJK Compatibility Ideograph`

쉽게 말하면:
- 한자처럼 보이는 글자만 뽑는다.

비포함:
- 한글
- 영어
- 숫자
- 공백
- 기호
- 이모지

---

## 중복 판정 기준

중복 기준은 사용자가 정한 값으로 계산한다.

- 기본값은 `2번 이상`
- 사용자가 popup에서 이 값을 바꿀 수 있다.
- 마지막으로 사용한 최소 반복 횟수는 저장해서 다음 실행 때 그대로 보여준다.

예:
- `刊`이 전체 제목에서 `3번` 나옴 -> 포함
- `再`가 전체 제목에서 `2번` 나옴 -> 포함
- `龘`이 `1번`만 나옴 -> 제외

예를 들어 최소 반복 횟수를 `5`로 바꾸면:
- `5번 이상` 나온 글자만 결과에 남는다.

즉 결과 문자열에는 **설정한 최소 반복 횟수 이상인 글자만** 들어간다.

## 출력 형식

### 1. 메인 결과

긴 결과 로그 영역에 표시되는 한 줄 텍스트:

```text
刊,扺,縐,倻,偶,再,...
```

규칙:
- 한 글자씩
- `,` 로 구분
- 중복 한자만 포함
- **등장 횟수 내림차순**으로 정렬
- **쉼표 뒤 공백 없음**
- 결과는 popup의 작은 입력칸이 아니라 **길게 볼 수 있는 로그/결과 영역**에 출력

### 2. 보조 통계

같이 보여주면 좋은 값:
- 분석 페이지 수
- 분석한 제목 수
- 제목에서 추출한 전체 한자 수
- 고유 한자 수
- 중복 한자 종류 수

예:

```text
분석 페이지: 400
분석 제목 수: 5231
전체 한자 수: 41848
고유 한자 수: 9720
중복 한자 종류 수: 611
```

### 3. 선택 출력

추후 있으면 좋은 것:
- 상위 빈도 한자 표
- `글자: 횟수`
- CSV 복사
- TXT 저장

하지만 1차 구현 필수는 아니다.

## 수집 방식

### 1차 구현 목표

현재 열려 있는 관리기록 페이지 기준으로:
- 확장이 페이지 DOM을 읽는다
- 다음 페이지로 이동한다
- 다시 DOM을 읽는다
- 끝 페이지까지 반복한다

즉 “HTML 파일 업로드”가 아니라:
- **실제 관리기록 페이지를 직접 순회하는 방식**
을 1차 목표로 한다.

### 실제 HTML 기준 확정된 수집 규칙

받은 관리기록 HTML 기준으로 1차 구현은 아래 selector와 규칙으로 고정한다.

- 페이지 URL 형식:
  - `https://gall.dcinside.com/mgallery/management/block?id=thesingularity&s=&t=u&p=1`
  - 즉 페이지 이동은 `p=` 값만 바꿔서 한다.
- row selector:
  - `tbody > tr`
- 종류 판별 selector:
  - `td.blockcontent em`
- 제목 selector:
  - `td.blockcontent a`

즉 실제 파싱 순서는 아래처럼 고정한다.

1. 현재 페이지에서 `tbody > tr`를 전부 찾는다.
2. 각 row의 `td.blockcontent em` 텍스트를 읽는다.
3. 이 값이 정확히 `게시글`인 row만 남긴다.
4. `댓글` row는 버린다.
5. 남은 row에서 `td.blockcontent a` 텍스트를 제목으로 사용한다.
6. 제목에서 Han/CJK 글자만 한 글자씩 뽑는다.

중요:
- `tbody`를 그냥 전부 읽기만 하면 안 된다.
- 이유는 `댓글` row도 구조가 거의 같기 때문이다.
- 그래서 반드시 `em === 게시글` 필터를 먼저 적용해야 한다.

- 시작 페이지 입력
- 끝 페이지 입력

예시 상태 표시:
- `12 / 400 페이지 분석 중`
- `총 제목 318개 수집`

```text
刊,扺,縐,倻,偶,再,...
```

즉 결과 표시는:
- 짧은 한 줄 입력칸이 아니라
- **스크롤 가능한 긴 로그 영역**
에 보여줘야 한다.

## 1차 구현 범위

1차는 아래까지만 해도 충분하다.

- 관리기록 페이지 `1~400` 순회
- `게시글` 제목만 추출
- 한자/CJK 글자만 추출
- 중복 글자만 결과 출력
- 복사 가능

1차에서 굳이 안 해도 되는 것:
- 닉/IP 분석
- 제목 단위 중복 분석
- 한자 빈도 히트맵
- 통신사 통계
- 공격 세션 비교 분석

## 핵심 요약

이 도구는:

- 관리기록 여러 페이지를 자동으로 읽고
- 제목에서 한자만 한 글자씩 뽑은 뒤
- 중복되는 한자만 추려서
- `한자,한자,한자,...`
형태로 보여주는 보조 확장이다.

가장 중요한 요구사항은 딱 4개다.

1. **제목만 본다**
2. **한자 한 글자씩 뽑는다**
3. **여러 페이지를 순회한다**
4. **설정한 최소 반복 횟수 이상인 한자만 쉼표로 이어 출력한다**

## 구현 전에 필요한 기술 스펙

위 요구사항은 확정됐고, 이제 구현하려면 아래 **HTML/페이지 구조 스펙**이 필요하다.

### 꼭 필요한 스펙

1. **관리기록 페이지 URL 형식**
- 확인 완료
- 실제 형식은 `...?p=1`
- 즉 페이지 번호는 URL query `p`로 직접 바꿀 수 있다.

2. **관리기록 테이블 row selector**
- 확인 완료
- 실제 row selector는 `tbody > tr`

3. **`게시글` 행 판별 기준**
- 확인 완료
- `td.blockcontent em` 텍스트로 판별
- 값이 `게시글`이면 포함
- 값이 `댓글`이면 제외

4. **제목 셀 selector**
- 확인 완료
- 제목은 `td.blockcontent a`
- 제목 텍스트는 `a.textContent` 사용

5. **다음 페이지 이동 방식**
- 확인 완료
- `p=` 값만 바꿔 URL 직접 이동

6. **로딩 완료 판정**
- 어떤 시점에 “이 페이지 파싱 가능”이라고 볼지
- 예: row 1개 이상 보임
- 또는 특정 테이블 selector 등장

7. **마지막 페이지 / 빈 페이지 판정**
- 빈 row면 종료할지
- `페이지 없음` 문구가 있는지
- pagination에 마지막 번호가 보이는지

### 있으면 좋은 스펙

8. **관리기록 화면의 실제 HTML 예시 1페이지**
- 확인 완료

9. **400페이지 같은 뒤쪽 실제 HTML 예시**
- 구조가 같은지 확인용

10. **로그인 유지 방식**
- 확장이 현재 로그인된 탭에서만 돌면 되는지
- 새 탭 이동 시에도 같은 세션이 유지되는지

### 중요한 판단

1차 구현은 **API 스펙이 없어도 된다.**

이유:
- 이번 도구는 관리기록 페이지를 직접 순회하면서
- 현재 보이는 HTML에서 제목을 읽기만 하면 되기 때문이다.

즉 지금 필요한 건:
- **API 스펙보다 HTML/DOM 스펙**
- 특히 `row`, `게시글 판별`, `제목 위치`, `페이지 이동 방식`
이 4개다.

현재 기준으로는 이 4개 중 아래는 이미 확정되었다.

- row: `tbody > tr`
- 게시글 판별: `td.blockcontent em === '게시글'`
- 제목 위치: `td.blockcontent a`
- 페이지 이동: `p=` query 변경
