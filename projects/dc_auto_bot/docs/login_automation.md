# 로그인 세션 자동화 스펙

## ownership 메모

이 문서는 **신문고봇 내부에 현재 구현돼 있는 로그인 자동화의 레퍼런스 문서**다.

운영 구조 변경 기준으로는:

- 세션 유지와 계정1/계정2 전환의 최종 ownership은 특궁 메인 확장의 `dcSessionBroker`로 이전하는 방향이 맞다
- 신문고봇은 장기적으로 로그인 자동화 dependency를 제거하고 공유 쿠키 세션만 사용하는 쪽으로 정리된다

관련 문서:

- [dc_session_broker_login_keepalive_plan.md](../../../docs/dc_session_broker_login_keepalive_plan.md)
- [login_automation_dependency_removal_plan.md](./login_automation_dependency_removal_plan.md)

## 목적

신문고 봇은 장시간 무인 운영 중 디시인사이드 로그인 세션 또는 매니저 권한 세션이 풀리면 실제 삭제/차단 단계에서 실패할 수 있다.

이 문서의 목적은 다음을 정의하는 것이다.

1. 로그인 세션이 풀렸는지 주기적으로 감지하는 방법
2. 로그아웃 상태일 때 자동 로그인하는 방법
3. 2차 인증, 캡차, 사이트 구조 변경 등 자동화 불가 상황을 UI에서 경고하는 방법
4. 사용자 일반 탭을 건드리지 않고 전용 체크 탭에서만 세션 확인을 수행하는 방법


## 배경

현재 봇은 다음 단계가 분리되어 있다.

1. 신뢰 사용자 댓글 명령 수신
2. 작성자 필터 / 최근 100개 / 개념글 가드
3. Gemini 판정
4. 실제 삭제/차단 AJAX 실행

이 중 4번은 디시 로그인 세션 + 매니저 권한 세션이 살아 있어야만 성공한다.

세션이 풀리면 다음과 같은 오류가 난다.

- `정상적인 접근이 아닙니다.`
- `관리권한이 없습니다.`
- `ci_t 토큰(ci_c 쿠키)을 찾지 못했습니다.`

또한 실제 운영 경험상, 디시 탭이 열려 있어도 **새로고침 전에는 로그인 상태처럼 보이다가 새로고침 후 로그아웃 상태가 드러나는 경우**가 있기 때문에, 로그인 상태 판정은 반드시 **새로고침 기반**으로 해야 한다.


## 범위

이 문서는 **신문고 봇 확장 프로그램 내부의 디시 로그인 세션 자동화**만 다룬다.

- 포함:
  - 로그인 상태 감지
  - 자동 로그인
  - UI 상태 표기
  - 재시도 정책
  - 전용 세션 체크 탭 관리
- 제외:
  - helper/Gemini 로그인
  - Cloudflare Tunnel
  - 공개 transparency 사이트 배포


## 확정 정책

### 1. UI 구조

popup UI는 다음처럼 변경한다.

#### Helper 상태 카드

기존 `Helper 상태` 카드 안에 다음 두 상태를 같이 표시한다.

- helper 연결 상태
  - 예: `🟢 helper 연결 정상`
  - 예: `🔴 helper 연결실패`
- login 연결 상태
  - 예: `🟢 login 연결 정상`
  - 예: `🔴 login 연결실패`

주의:
- `login 연결 상태`는 실제로는 디시 세션 상태이지만, 사용자 입장에서는 운영 필수 의존성이라 `Helper 상태` 카드 안에 함께 표기한다.

#### 봇 제어 카드

기존 `봇 제어` ON/OFF 토글 아래에 작은 보조 토글을 추가한다.

- `로그인 세션 자동화`

의미:
- OFF: 세션 확인/자동 로그인 수행 안 함
- ON: 전용 체크 탭에서 주기적 세션 확인 및 자동 로그인 수행


### 2. 계정 입력 방식

아이디/비밀번호는 확장 popup 내부 설정 UI에서 사용자가 직접 입력한다.

- `디시 아이디`
- `디시 비밀번호`

비밀번호는 다음 정책으로 처리한다.

- UI에서는 `password` 타입 입력 필드로 마스킹한다
- 별도 암호화 UX 요구는 두지 않는다
- 사용자가 로컬 PC에서만 쓰는 전제를 둔다

구현 메모:
- 브라우저 재시작/확장 새로고침 후에도 유지되어야 하므로 실제 값은 확장 로컬 저장소에 저장해야 한다
- 화면에 다시 보여줄 때는 마스킹만 적용한다


### 3. 로그인 상태 확인 방식

로그인 상태 확인은 **30초 주기 새로고침 방식**으로 한다.

중요 이유:
- 디시 페이지는 새로고침 전까진 로그인된 것처럼 보일 수 있다
- 따라서 단순 DOM 확인이 아니라 **새로고침 후 DOM 기준**으로 상태를 판정해야 한다


### 4. 전용 세션 체크 탭

사용자 일반 탭을 건드리지 않기 위해, 확장 프로그램이 관리하는 **전용 세션 체크 탭 1개**를 둔다.

정책:
- `로그인 세션 자동화` 토글이 ON이면 생성/유지한다
- 봇이 OFF여도 `로그인 세션 자동화` 토글이 ON이면 유지한다
- `active: false`로 열어 사용자가 보던 흐름을 방해하지 않는다
- 가능하면 pinned 처리한다
- URL은 항상 특갤 목록 페이지로 유지한다
- 탭이 닫히면 background가 다시 생성한다
- 여러 개를 만들지 않고 단일 탭만 유지한다

세션 확인은 **이 전용 탭에서만** 수행한다.

즉 사용자가 보고 있는 일반 디시 탭을 30초마다 새로고침하지 않는다.


### 5. 로그인 상태 판정 기준

전용 체크 탭을 새로고침한 뒤, 특갤 페이지 HTML에 아래 버튼이 존재하면 **로그인 성공 + 매니저 권한 성공**으로 본다.

```html
<button type="button" class="sp_img btn_useradmin_go" onclick="location.href='/mgallery/management?id=thesingularity';"><span class="blind">관리</span></button>
```

즉 성공 판정 기준은 단순 로그인 여부가 아니라:

1. 해당 페이지에 접근 가능하고
2. 특갤 매니저 권한 UI가 뜨는가

이다.

이 버튼이 없으면 다음 둘 중 하나로 본다.

- 로그아웃 상태
- 로그인은 되어 있으나 특갤 매니저 권한이 없는 상태

둘 다 봇 운영 관점에서는 `login 연결실패`로 처리한다.

로그아웃 상태는 다음 로그인 유도 요소가 존재할 때로 본다.

```html
<strong onclick="location='https://sign.dcinside.com/login?s_url=https%3A%2F%2Fgall.dcinside.com%2Fmgallery%2Fboard%2Flists%2F%3Fid%3Dthesingularity&s_key=686'" style="cursor:pointer">로그인해 주세요.</strong>
```

즉 특갤 목록 페이지를 새로고침한 뒤 위 요소가 보이면 내부 상태는 `logged_out`이다.

잘못된 계정 / 매니저 권한 없음 상태는 다음처럼 본다.

- `btn_useradmin_go` 없음
- 로그인 유도 요소도 없음

이 경우 내부 상태는 `wrong_account_or_no_manager`이고, UI에는 `login 연결실패`와 권한 없음 경고를 띄운다.


### 6. 자동 로그인 트리거

자동 로그인은 다음 조건을 모두 만족할 때만 시도한다.

1. `로그인 세션 자동화` 토글이 ON
2. 세션 체크 탭 새로고침 후 내부 상태가 `logged_out`
3. 현재 자동 로그인 시도가 진행 중이 아님

즉 **로그아웃 상태일 때만** 자동 로그인 시도한다.

주의:
- `wrong_account_or_no_manager` 상태에서는 자동 로그인하지 않는다
- 이 상태는 수동 확인 대상이다


### 7. 자동 로그인 동작

전용 세션 체크 탭에서 다음 순서로 수행한다.

1. 로그인 페이지 또는 로그인 가능한 페이지로 이동
2. popup에 저장된 아이디/비밀번호 입력
3. 로그인 버튼 제출
4. 특갤 페이지로 다시 이동
5. 새로고침 후 `btn_useradmin_go` 존재 여부 확인

성공 조건:
- 특갤 페이지 새로고침 후 `btn_useradmin_go`가 존재

실패 조건:
- 로그인 폼이 계속 남아 있음
- 로그인 오류 문구 존재
- 2차 인증 또는 캡차가 뜸
- 페이지 구조가 달라 selector를 찾지 못함

로그인 페이지 기본 selector는 다음으로 확정한다.

- 아이디 input
  - 1차: `#id`
  - 2차: `input[name="user_id"]`
- 비밀번호 input
  - 1차: `#pw`
  - 2차: `input[name="pw"]`
- 로그인 버튼
  - 1차: `button[type="submit"]`
  - 2차: `button.btn_blue.small.btn_wfull`

확인된 로그인 페이지 DOM 예시는 다음과 같다.

```html
<input type="text" id="id" name="user_id" value="" class="int id bg" title="식별 코드 입력" placeholder="식별 코드" maxlength="20">
<input type="password" id="pw" name="pw" class="int pw bg" title="비밀번호 입력" placeholder="비밀번호" maxlength="40">
<button type="submit" class="btn_blue small btn_wfull">로그인</button>
```


### 8. 2차 인증 / 캡차 / 사이트 변경 대응

다음 경우에는 자동 로그인을 더 진행하지 않고 UI 경고만 띄운다.

- 2차 인증 화면 등장
- 캡차 등장
- 로그인 페이지 구조가 바뀌어 selector를 찾지 못함

UI 표기:
- `🔴 login 연결실패`
- 보조 메시지 예:
  - `2차 인증 확인 필요`
  - `캡차 감지`
  - `로그인 페이지 구조 변경 가능`

현재 단계 정책:
- 2차 인증 / 캡차 / 페이지 변경은 세부 자동 처리 없이 **UI 경고만 띄우는 것**으로 충분하다
- 해당 상태를 만나면 자동 로그인 절차를 즉시 중단한다
- 세부 selector/문구는 실제 문제가 생겼을 때 추가 수집 후 보강한다


### 8.1 로그인 실패 판정

아이디/비밀번호가 틀린 경우의 실패 판정은 다음으로 확정한다.

- 로그인 제출 후 최종 URL이 `https://sign.dcinside.com/login/member_check` 이면 로그인 실패로 본다
- 페이지에서 다음 경고 문구가 확인되면 로그인 실패로 본다
  - `식별 코드 또는 비밀번호를 확인해 주세요.`

이 상태는 내부적으로 `credentials_invalid`로 취급한다.

현재 단계 구현 원칙:
- 경고 문구를 직접 읽지 못하더라도, 제출 후 `member_check` URL로 남아 있으면 로그인 실패로 처리한다
- 가능하면 경고 문구를 같이 수집해 `loginHealthMessage`에 반영한다


### 9. 재시도 정책

자동 로그인은 최대 3회까지 재시도한다.

정책:
- 기본 재시도 간격은 약 1분
- 각 재시도마다 지터를 둔다
- 확정값: `1분 ± 20초`

3회 모두 실패하면:
- 자동 로그인 시도 중단
- `10분 cooldown` 동안 추가 자동 로그인 시도를 하지 않는다
- UI에 경고 유지


### 10. 삭제 직전 / 실행 실패 시 재검사

30초 주기 체크와 별개로 다음 경우에는 즉시 세션 재검사를 수행한다.

1. 실제 삭제 실행 직전 1회
2. 삭제 API 응답이 아래 중 하나일 때
   - `정상적인 접근이 아닙니다.`
   - `관리권한이 없습니다.`

즉 운영 중 세션이 막 방금 풀린 경우도 빠르게 감지한다.


### 11. 자동 로그인 실패 후 봇 동작

자동 로그인 3회 실패 후에는 다음처럼 처리한다.

- 봇 본체를 자동 OFF 하지 않는다
- `login 연결실패` 경고 상태를 유지한다
- `10분 cooldown` 동안 자동 로그인 재시도를 중단한다

이유:
- 같은 브라우저 프로필에서 다른 자동화도 같이 돌 수 있다
- 신문고 봇 하나의 로그인 복구 실패만으로 전체 운영 흐름을 강하게 끊는 것은 과할 수 있다


### 12. 알림 방식

로그인 세션 자동화 실패/중단 시 알림 방식은 다음으로 확정한다.

- popup UI 상태 경고 표시
- 브라우저 notification 표시
- 자동 탭 포커스는 하지 않는다

이유:
- popup만으로는 사용자가 늦게 볼 수 있다
- 자동 포커스는 사용자 작업을 방해할 수 있다


### 13. 운영 중 상태 변화

상태는 다음처럼 관리한다.

- `login 연결 정상`
  - 세션 체크 성공
  - 매니저 버튼 확인됨
- `login 연결실패`
  - 로그인 풀림
  - 권한 없음
  - 2FA/캡차/사이트 변경
  - 자동 로그인 3회 실패
- `login 자동화 진행 중`
  - 자동 로그인 시도 중

UI에서는 최소한 다음을 보여준다.

- 현재 login 상태
- 최근 확인 시각
- 최근 자동 로그인 결과 메시지


## 구현 메모

### 상태 저장

확장 내부에서 다음 상태를 관리한다.

- `loginAutomationEnabled`
- `dcLoginUserId`
- `dcLoginPassword`
- `loginHealthStatus`
- `loginHealthMessage`
- `lastLoginCheckAt`
- `lastLoginAttemptAt`
- `loginRetryCount`
- `sessionCheckTabId`

전용 세션 체크 탭의 기본 URL은 다음으로 고정한다.

- `https://gall.dcinside.com/mgallery/board/lists/?id=thesingularity`


### 전용 체크 탭 URL

전용 체크 탭은 특갤 메인 또는 특갤 관리 진입이 가능한 페이지를 사용한다.

추천 URL:

- `https://gall.dcinside.com/mgallery/board/lists/?id=thesingularity`

이유:
- 매니저 버튼 존재 여부를 확인하기 쉽다
- 새로고침 시 로그인/권한 상태가 드러난다


### 봇 동작과의 관계

삭제 실행 직전에도 세션 상태가 정상인지 확인하는 보조 가드를 둘 수 있다.

다만 기본 정책은:
- 30초 주기 세션 체크
- 실패 시 자동 로그인
- 복구되면 정상 운영

으로 한다.


## 미정 정책

아래 항목은 방향은 잡혔지만 아직 정확한 구현 세부값이 미정이다.

주의:
- 재시도 횟수 / 지터 / cooldown은 확정됐다
- 자동 로그인 3회 실패 후 봇 동작은 확정됐다
- 브라우저 notification 경고 정책도 확정됐다
- 로그인 실패 판정 URL과 경고 문구는 확정됐다
- 현재 남은 핵심 미정은 2차 인증/캡차/페이지 변경을 서로 구분해서 표시할 세부 selector 여부다

### 1. 2차 인증 / 캡차 / 사이트 변경 감지 기준

아직 확정해야 하는 값:
- 2차 인증 화면 판단 selector 또는 문구
- 캡차 화면 판단 selector 또는 문구
- 성공 selector도 없고 로그인 폼 selector도 없을 때 `page_changed`로 보는 세부 기준

주의:
- 현재 정책상 이 항목들은 **세밀한 자동화보다 UI 경고 우선**이다
- 1차 구현에서는 세 항목을 서로 구분하지 않고, 전부 `manual_attention_required` 성격의 경고 상태로 묶어 처리한다
- 즉 자동 로그인 절차를 즉시 중단하고, `login 연결실패`와 경고 메시지만 표시하면 된다
- 실제 selector/문구 수집은 나중에 문제가 생겼을 때 보강한다


## 권장 구현 순서

1. popup UI에 `로그인 세션 자동화` 토글 추가
2. popup UI에 `디시 아이디 / 비밀번호` 입력 필드 추가
3. background에 전용 세션 체크 탭 생성/복구 로직 추가
4. 30초 주기 새로고침 + 로그인 유도 요소 / `btn_useradmin_go` 체크 구현
5. `login 연결 정상 / 실패` 상태 표시 추가
6. 자동 로그인 시도 로직 구현
7. 2FA/캡차/사이트 변경 감지 시 경고 처리 추가
8. 3회 재시도 + `1분 ± 20초` 지터 + `10분 cooldown` 적용


## 한 줄 요약

로그인 세션 자동화는 **전용 체크 탭 1개를 background가 관리하면서 30초마다 특갤 목록을 새로고침하고, 로그인 유도 요소 / 매니저 버튼 존재 여부로 상태를 판정한 뒤, 로그아웃 상태일 때만 자동 로그인 3회 재시도를 수행하며, 실패 시에는 `1분 ± 20초` 지터와 `10분 cooldown`, 브라우저 notification 경고를 사용하는 구조**로 간다.
