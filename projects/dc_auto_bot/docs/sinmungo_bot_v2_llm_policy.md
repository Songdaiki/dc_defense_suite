# DC Auto Bot - 신문고 봇 v2 LLM 판정 정책

## 1. 목표

이 문서는 신문고 봇 v2에서 `삭제 전 LLM 판정 게이트`를 도입할 때 필요한 정책을 정리한다.

판정 결과를 공개하는 transparency 웹사이트 스펙은 아래 문서를 따른다.

- [신문고 봇 v2 투명성 웹사이트 스펙](./sinmungo_bot_v2_transparency_site_spec.md)

v2 LLM의 역할은 다음 하나다.

- 신뢰 사용자가 신고한 대상 게시물이 **정말 신고 사유와 운영 규정에 맞는 악성 게시물인지**를 최종 판정한다.

즉 LLM은 명령 파서가 아니라 **삭제 실행 전 판단기**다.

## 2. 전제

LLM은 아래 조건을 모두 통과한 뒤에만 호출한다.

1. 신뢰 사용자 `user_id` 검증 통과
2. 명령 문법 파싱 통과
3. 링크 dedupe 통과
4. 일일 2회 제한 통과
5. v2 core 작성자 필터 통과
   - 유동 또는 깡계(`< 100`) 또는 글편중(`글 비중 >= 0.9`)
6. 대상 게시물이 `전체글 기준 최근 100개 regular row` 안에 있음
7. 대상 게시물이 개념글이 아님

즉 LLM은 전체 명령이 아니라 **이미 1차 운영 정책을 통과한 대상**에만 쓴다.

여기서 `최근 100개 regular row`는 다음처럼 정의한다.

- `전체글` 목록을 최신순으로 읽는다.
- `공지`, `설문`, 숫자 아닌 `gall_num` row는 제외한다.
- 제외 후 남는 일반 사용자 게시물 row를 `regular row`로 본다.
- 이 `regular row` 최신 100건만 자동 삭제 후보로 본다.
- 개념글 row는 recent 100 계산에서도 제외한다.

개념글 판정은 다음처럼 정의한다.

- 대상 게시물 view HTML의 `#recommend` hidden input 값이 `K`면 개념글로 본다.
- 개념글이면 helper 결과와 무관하게 자동 삭제/차단하지 않는다.
- 위 판정이 실패하면 fail-safe로 자동 실행하지 않는다.

## 3. 모델/인증 방식

v2 LLM은 **Gemini CLI + Google OAuth**를 전제로 한다.

즉 구조는 다음과 같다.

- 확장 프로그램
- 로컬 relay/helper 프로세스
- Gemini CLI (OAuth 로그인 상태)
- Gemini 모델 응답

중요:
- 이번 단계에서는 유료 Gemini API를 기본 전제로 하지 않는다.
- 우선은 **Gemini CLI OAuth 세션으로 직접 테스트 가능한 구조**를 목표로 한다.
- 따라서 운영자가 링크 몇 개를 직접 넣어 `allow / confidence`가 어떻게 나오는지 먼저 검증할 수 있어야 한다.

## 4. 기본 원칙

운영 규정 `P1 ~ P15`는 **LLM이 직접 판정**한다.

즉 다음처럼 나누지 않는다.

- 일부는 정적 규칙으로 먼저 처리
- 일부는 LLM으로만 처리

이 문서 기준으로는 **정책 ID 전체를 LLM이 직접 매칭**한다.

단, 비용/속도/오판 리스크를 줄이기 위해 다음 원칙은 유지한다.

1. 같은 `postNo`는 dedupe되어 LLM도 한 번만 호출한다.
2. 본문 전체 HTML이 아니라 **정제된 제목/본문/이미지 정보**만 보낸다.
3. 이미지 입력 수준은 단계적으로 올린다.
4. 처음 단계는 자동 삭제가 아니라 **수동 테스트 모드**를 우선한다.

## 5. 운영 규정 구조화

운영 규정의 **원문 source of truth**는 아래 문서다.

- [특이점 갤러리 운영 방침 원문](./thesingularity_gallery_policy.md)

현재 v2 구현에서는:

- helper 프롬프트에 위 원문을 직접 포함한다.
- 아래 `P1 ~ P15`는 원문을 LLM 판정용 ID로 구조화한 인덱스다.
- 즉 아래 구조화 표는 **원문을 대체하는 축약본이 아니라, 원문과 함께 쓰는 매핑표**다.

### 5.1 정책 ID

- `P1`: 디시 이용약관, 법률, 건전한 사회 통념 위반
  - 액션: 글 삭제, 30일 차단
- `P2`: 닉언, 친목질, 사칭
  - 액션: 글 삭제, 30일 차단
- `P3`: 분탕 및 어그로, 꼬투리 잡기, 고로시
  - 액션: 글 삭제, 30일 차단
- `P4`: 종교, 음모론, 자연의 섭리 포함
  - 액션: 글 삭제, 30일 차단
- `P5`: 반과학, 유사과학, 반지성주의, 직업 비하/조롱
  - 액션: 글 삭제, 30일 차단
- `P6`: 레퍼런스 없는 선형글
  - 액션: 글 삭제, 30일 차단
- `P7`: 주제 무관, 일침/설교성 글
  - 액션: 글 삭제, 30일 차단
- `P8`: 무인증 현직자/전공자 주장, 과도한 팬보이, 갈드컵 유발
  - 액션: 글 삭제, 7~30일 차단
- `P9`: 주식, 코인, 투자 관련 글
  - 액션: 글 삭제, 7일 차단
- `P10`: 과도한 국뽕/일뽕/중뽕, 출산율, 혐한/국까 떡밥
  - 액션: 글 삭제, 7일 차단
- `P11`: 정치인/정당/정책/정치사상/지역드립/성별 혐오
  - 액션: 글 삭제, 7일 차단
- `P12`: 타 갤러리, 타 커뮤니티 언급
  - 액션: 글 삭제, 1일 차단
- `P13`: 맥락 없는 시비성 욕설, 상호 욕설 싸움
  - 액션: 글 삭제, 1일 차단
- `P14`: 금지 떡밥 게시글
  - 액션: 글 삭제
- `P15`: 개념글 제한 대상
  - 액션: 개념글 제한

### 5.2 LLM 판정 범위

LLM은 `P1 ~ P15` 전체를 판정 대상으로 본다.

즉 아래를 모두 LLM이 직접 판단한다.

- 어떤 정책 ID에 해당하는지
- 자동 삭제/차단 게이트를 열어도 되는지
- 애매하면 review로 넘겨야 하는지

## 6. 입력 데이터 정책

LLM에 넘기는 입력은 기본적으로 아래를 사용한다.

- 갤러리 ID
- 대상 게시물 URL
- 대상 게시물 제목
- 대상 게시물 본문 텍스트
- 대상 게시물 첨부 이미지 정보
- 신문고 댓글의 `사유:` 뒤 텍스트
- 요청자 `label`
- v2 core 작성자 판정 결과
  - `fluid` | `low_activity` | `normal`

### 6.1 제목 / 본문 파싱 정책

LLM 호출 전 본문은 HTML 그대로 주지 않는다.

정제 대상:
- 제목 텍스트
- 본문 텍스트
- 본문 내 외부 링크 목록
- 첨부 이미지 URL 목록

권장 파서 범위:
- 제목: `.title_subject`
- 본문: `.writing_view_box .write_div`
- 이미지: `.writing_view_box img`

정제 원칙:
- HTML 태그 제거
- 연속 공백 축약
- 너무 긴 본문은 길이 제한
- 불필요한 갤 UI 문구 제거

즉 LLM에는 `사람이 읽는 게시물 본문 요약 입력`만 들어가야 한다.

### 6.2 이미지 정책

사용자 요구 기준으로 v2는 **제목 / 본문 / 이미지**를 모두 LLM 판단 입력으로 쓰는 방향이다.

현재 단계의 이미지 정책은 **`v2.0 URL-only`로 고정**한다.

- 제목 + 본문 + 이미지 URL 목록만 LLM 입력으로 사용한다.
- OCR 텍스트 추출은 이번 범위에 포함하지 않는다.
- 실제 이미지 vision 입력도 이번 범위에 포함하지 않는다.

즉 현재 구현 범위에서 이미지는 **URL 전달 전용**이다.

## 7. 출력 형식 정책

LLM 출력은 자유문이 아니라 **고정 JSON**이어야 한다.

```json
{
  "decision": "allow" | "deny" | "review",
  "confidence": 0.0,
  "reason": "짧은 판정 사유",
  "policy_ids": ["P3", "P14"]
}
```

의미:
- `allow`
  - 운영 규정 위반이 명확하여 자동 삭제 + 차단을 진행해도 되는 경우
- `deny`
  - 운영 규정 위반이 아니거나, 자동 삭제를 진행하면 안 되는 경우
- `review`
  - 위반 가능성은 있으나 자동 삭제를 바로 진행하지 않고 운영진 검토가 필요한 경우

추가 강제 규칙:
- `policy_ids = ["NONE"]` 이면 `decision`은 반드시 `deny`여야 한다.
- `allow`는 최소 1개 이상의 정책 위반(`P1 ~ P15`)이 명확히 성립할 때만 사용한다.
- 위반 가능성이 있으나 확신이 부족하면 `review`를 사용한다.

## 8. 기본 실행 정책

기본 정책은 보수적으로 간다.

- 최근 100개 regular row 안에 있음
- 개념글 아님
- `decision = allow`
- `confidence >= 0.85`

이 네 조건을 모두 만족할 때만 실제 삭제 + 차단 실행.

그 외는 모두 자동 실행하지 않는다.

즉:
- 최근 100개 밖 -> 실행 안 함
- 개념글 -> 실행 안 함
- `deny` -> 실행 안 함
- `review` -> 실행 안 함
- `allow`여도 신뢰도 낮음 -> 실행 안 함
- `policy_ids=["NONE"]` -> 무조건 실행 안 함

## 9. Fail-safe 정책

아래 경우는 모두 **삭제 실행 금지**다.

- Gemini CLI 호출 실패
- OAuth 인증 만료
- relay/helper 타임아웃
- JSON 파싱 실패
- 필수 키 누락
- `decision` 값 이상
- `confidence` 누락 또는 숫자 아님
- `policy_ids` 누락

즉 오류 시 기본값은 항상 `실행 안 함`이다.

## 10. 비용 최소화 구현안

정적 규칙 선검사를 빼는 대신, 비용 최소화는 아래 방식으로 달성한다.

1. dedupe로 같은 게시물은 한 번만 Gemini에 보낸다.
2. 제목/본문은 정제된 텍스트만 전달한다.
3. 본문 길이는 일정 길이에서 잘라서 전달한다.
4. 이미지 입력은 `v2.0 URL-only`로 고정한다.
5. Gemini는 가능한 한 작은 텍스트 모델부터 시작한다.

즉 비용 관점에서 권장 순서는:
- `작은 텍스트 모델`
- `URL-only 이미지`
순이다.

## 11. 수동 테스트 우선 정책

이번 단계에서는 자동 shadow mode보다 **운영자가 링크를 직접 넣어 테스트하는 수동 테스트 모드**를 우선한다.

즉 구현 첫 단계는 다음과 같다.

1. 운영자가 확장 또는 별도 테스트 UI에 대상 링크를 입력
2. 대상 게시물 제목/본문/이미지 정보를 파싱
3. Gemini CLI OAuth 세션으로 판정 요청
4. 결과 JSON을 화면/로그에 표시
   - `decision`
   - `confidence`
   - `policy_ids`
   - `reason`
5. 실제 삭제는 하지 않음

수동 테스트의 목적은:

- 판단 결과가 일관적인지 확인
- `allow / deny / review`가 기대대로 나오는지 확인
- 실제 운영 연결 전에 오탐/누락 감을 보는 것

수동 테스트 단계에서는 최근 100개 / 개념글 자동 삭제 금지 가드를 적용하지 않는다.

즉 수동 테스트는 **판단 잘하는지 여부만 확인하는 검증 단계**다.

이 방식의 장점:
- 운영진이 링크 몇 개로 바로 성능을 체감할 수 있음
- 비용을 예측하기 쉬움
- 프롬프트 튜닝이 쉬움
- 잘못된 자동 삭제를 막을 수 있음

## 12. Shadow mode 권장

수동 테스트가 어느 정도 안정화되면 그 다음 단계로 shadow mode를 권장한다.

- 신뢰 사용자 명령 수신
- v2 core 작성자 필터 통과
- 최근 100개 regular row / 개념글 여부 판정
- LLM 호출
- 결과 로그만 기록
- 실제 삭제/차단은 하지 않음

이렇게 하면:
- 악성 판정률
- 오탐률
- 응답 속도
- 운영진 신뢰도
- 비용

을 실제 데이터로 볼 수 있다.

## 13. 구현 순서 권장안

1. v2 core 작성자 필터 먼저 구현
2. 제목/본문/이미지 URL 파서 구현
3. Gemini CLI OAuth relay/helper 준비
4. 수동 테스트 모드 구현
5. 로그 기준으로 프롬프트/임계치 튜닝
6. 필요하면 shadow mode 추가
7. 충분히 맞는다고 판단되면 `allow + confidence >= threshold`에서만 실제 실행

## 14. 구현 전 최종 확인 필요 사항

- Gemini CLI를 어떤 방식으로 relay/helper에서 호출할지
- helper가 로컬 프로세스인지 간단한 로컬 서버인지
- confidence threshold를 몇으로 둘지
- 수동 테스트 UI를 확장 안에 넣을지, 별도 스크립트로 둘지

현재 단계에선 이 문서를 기준으로 **정책 설계와 비용 절감 방향까지는 고정 가능**하고, 실제 구현은 Gemini CLI helper 방식이 확정되면 들어가는 것이 맞다.


## 15. 확장 직접 OAuth + 직접 Gemini 호출 POC

v2는 helper/relay 없이 **확장 프로그램에서 직접 Google OAuth 토큰을 받고, Gemini를 직접 호출하는 방향**도 검토한다.

### 15.1 목표

POC 목표는 두 가지다.

1. 확장에서 Google OAuth 토큰을 직접 받을 수 있는지 확인
2. 그 토큰으로 Gemini 표준 호출이 실제로 붙는지 확인

### 15.2 POC 단계

1. 확장에 `Google 로그인` 버튼 추가
2. Chrome Identity API 또는 OAuth web flow로 access token 획득
3. `텍스트 전용 테스트` 호출
   - 제목 + 본문 + 신고 사유만 전달
4. `이미지 포함 테스트` 호출
   - `v2.0`: 이미지 URL만 프롬프트에 포함
5. 결과를 popup/log에 그대로 표시
   - `decision`
   - `confidence`
   - `policy_ids`
   - `reason`

### 15.3 이미지 전달 단계별 확인

현재 POC 기준 이미지 정책도 **`v2.0 URL-only`로 고정**한다.

1. `텍스트만` 전달
   - OAuth + Gemini 직접 호출이 붙는지 확인
2. `이미지 URL만` 포함
   - 제목/본문과 함께 URL 목록 전달

`이미지 blob/base64` 직접 전달은 이번 범위에 포함하지 않는다.

### 15.4 필요한 스펙

직접 구현 전에 아래를 확정해야 한다.

- 확장에서 쓸 Google OAuth 방식
- 필요한 scope
- Gemini 호출 endpoint
- 확장에서 직접 호출 시 CORS/host permission 필요 범위
- 이미지는 URL 목록만 전달

### 15.5 성공 기준

아래가 되면 POC 성공으로 본다.

1. 확장에서 로그인 버튼으로 토큰 획득 가능
2. 제목/본문 텍스트를 Gemini로 보내 JSON 응답 수신 가능
3. 이미지 URL 포함 테스트 결과를 같은 JSON 형식으로 수신 가능
4. `NONE -> deny`, `allow -> 실제 위반` 의미가 프롬프트 기준으로 안정적

### 15.6 실패 시 롤백 기준

아래 중 하나면 `확장 직접 Gemini 호출`을 롤백하고 다른 구조로 되돌린다.

- OAuth 토큰 획득 실패
- Gemini 직접 호출이 CORS/인증 문제로 불안정
- 이미지 URL 기반 판단이 너무 불안정
- JSON 출력이 자주 깨짐

즉 v2는 **붙여보고 안 되면 롤백**할 수 있게, POC를 작은 단계로 끊어서 진행하는 것이 원칙이다.
