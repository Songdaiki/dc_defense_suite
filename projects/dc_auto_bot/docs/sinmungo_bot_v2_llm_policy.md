# DC Auto Bot - 신문고 봇 v2 LLM 판정 정책

## 1. 목표

이 문서는 신문고 봇 v2에서 `삭제 전 LLM 판정 게이트`를 도입할 때 필요한 정책을 정리한다.

v2 LLM의 역할은 다음 하나다.

- 신뢰 사용자가 신고한 대상 게시물이 **정말 신고 사유에 맞는 악성 게시물인지**를 최종 판정한다.

즉 LLM은 `명령 파서`가 아니라 **삭제 실행 전 판단기**다.

## 2. 전제

LLM은 v1의 기본 조건을 모두 통과한 뒤에만 호출한다.

1. 신뢰 사용자 `user_id` 검증 통과
2. 명령 문법 파싱 통과
3. 링크 dedupe 통과
4. 일일 2회 제한 통과
5. v2 core 작성자 필터 통과
   - 유동 또는 깡계(`< 100`)

즉 LLM은 전체 명령이 아니라 **이미 1차 운영 정책을 통과한 대상**에만 쓴다.

## 3. 왜 별도 정책이 필요한가

LLM을 바로 삭제 액션에 연결하면 아래 리스크가 있다.

- 오판 시 정상 게시물 삭제
- 모델/프롬프트 변경에 따른 판정 흔들림
- 응답 지연으로 운영 처리 지연
- API 키/토큰 노출 문제
- 누가 왜 삭제했는지 감사 추적이 어려워짐

그래서 v2는 **판정 기준, 입력 데이터, 출력 형식, 실패 시 동작**을 문서로 먼저 고정해야 한다.

## 4. 입력 데이터 정책

LLM에 넘기는 입력은 다음으로 제한한다.

- 갤러리 ID
- 대상 게시물 URL
- 대상 게시물 제목
- 대상 게시물 본문 텍스트
- 신문고 댓글의 `사유:` 뒤 텍스트
- 요청자 `label`

기본적으로 v2 첫 단계에서는 다음은 제외한다.

- 댓글 전체
- 첨부 이미지 OCR
- 외부 링크 본문 fetch
- 사용자 프로필 정보 전체

이유는 비용/속도/오판 범위를 줄이기 위해서다.

## 5. 출력 형식 정책

LLM 출력은 자유문이 아니라 **고정 JSON**이어야 한다.

```json
{
  "decision": "allow" | "deny" | "review",
  "confidence": 0.0,
  "reason": "짧은 판정 사유",
  "matched_policy": ["spam", "ad", "malicious_gibberish"]
}
```

의미:

- `allow`
  - 자동 삭제 + 차단 진행 가능
- `deny`
  - 자동 처리 안 함
- `review`
  - 자동 처리 안 하고 로그/검토 대기

## 6. 기본 정책

v2 기본 정책은 보수적으로 간다.

- `decision = allow` 이고
- `confidence >= 0.85` 일 때만
- 실제 삭제 + 차단 실행

그 외는 모두 자동 실행하지 않는다.

즉:

- `deny` -> 실행 안 함
- `review` -> 실행 안 함
- `allow`여도 신뢰도 낮음 -> 실행 안 함

## 7. Fail-safe 정책

아래 경우는 모두 **삭제 실행 금지**다.

- LLM API 타임아웃
- 인증 실패
- JSON 파싱 실패
- 필수 키 누락
- `decision` 값이 정책 외 문자열
- `confidence` 누락 또는 숫자 아님

즉 오류 시 기본값은 항상 `실행 안 함`이다.

## 8. 인증 / 비밀키 정책

중요: 브라우저 확장에 LLM API 비밀키를 직접 넣는 방식은 권장하지 않는다.

권장 구조:

- 확장 -> 중간 relay/backend 호출
- backend -> 실제 LLM provider 호출

이유:

- 확장 코드에 키가 노출되기 쉬움
- 여러 운영진 PC에 키를 배포해야 함
- 사용량/로그/차단 정책을 서버에서 통제하기 어려움

즉 v2 LLM은 **직접 provider API를 브라우저에서 두드리는 방식보다 relay 서버 방식**이 기본안이다.

## 9. 로그 / 감사 정책

LLM 연동 시 아래를 로컬 로그 또는 서버 로그에 남긴다.

- 요청 시각
- 요청자 `label`
- 대상 게시물 번호
- 신문고 사유 원문
- LLM decision
- LLM confidence
- LLM reason
- 실제 삭제 실행 여부

즉 나중에 `누가`, `왜`, `어떤 근거로` 자동 삭제를 했는지 추적 가능해야 한다.

## 10. v2 구현 순서 권장안

1. v2 core 작성자 필터 먼저 구현
2. LLM은 `shadow mode`부터 시작
   - 실제 삭제는 하지 않고 판정만 로그로 남김
3. 로그를 보고 프롬프트/임계치 튜닝
4. 충분히 맞는다고 판단되면 `allow + confidence >= threshold`에서만 실제 실행

즉 LLM은 처음부터 바로 삭제에 물리지 않고, **판정 로그 모드 -> 검증 -> 제한적 실행** 순으로 가는 게 맞다.

## 11. Shadow mode 권장

v2 초기 테스트는 아래처럼 한다.

- 신뢰 사용자 명령 수신
- v2 core 작성자 필터 통과
- LLM 호출
- 결과 로그만 기록
- 실제 삭제/차단은 하지 않음

이렇게 하면:

- 악성 판정률
- 오탐률
- 응답 속도
- 운영진 신뢰도

를 실제 데이터로 볼 수 있다.

## 12. 구현 전 최종 확인 필요 사항

- 어떤 LLM provider를 쓸지
- relay/backend를 둘지
- 입력 데이터에 제목/본문 외에 댓글 일부까지 넣을지
- confidence threshold를 몇으로 둘지
- shadow mode를 얼마나 돌릴지

현재 단계에선 이 문서를 기준으로 **정책 설계와 테스트 계획까지는 가능**하고, 실제 구현은 provider/relay 결정 후 들어가는 것이 맞다.
